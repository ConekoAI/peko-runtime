# ADR-012: No first-party external channel integrations

**Date:** 2026-03-16
**Status:** Accepted
**Deciders:** Core team

## Context

Users naturally want to talk to their agents through familiar channels: Discord, Slack, Telegram, WhatsApp, email. Early feature discussions included building first-party integrations for these channels into Pekobot.

## Decision

Pekobot ships no first-party Discord bot, Slack app, Telegram integration, email handler, or any other external channel adapter. The daemon exposes a generic inbound webhook endpoint (`POST /webhooks/{instance_id}/{token}`) and a WebSocket endpoint. External channel integrations are third-party adapters built on these endpoints.

## Reasoning

**Channel maintenance is an unbounded commitment.** Discord changes its API. Slack introduces new scopes. WhatsApp adds new message types. Each channel integration requires ongoing maintenance to keep up with platform changes. Maintaining N integrations is N times the work, and breaking changes in any channel become Pekobot bugs. This is a poor use of core runtime engineering time.

**Channels are opinionated; Pekobot should not be.** Different users want different channels. An enterprise user wants Slack. A gamer wants Discord. A developer wants a raw WebSocket. A researcher wants email. Building first-party support for any channel means implicitly deprioritising others. The webhook API treats all channels equally.

**The webhook endpoint is sufficient.** Any channel that can POST to an HTTP endpoint can integrate with Pekobot. Most have webhook support. For channels that don't POST (they require polling or streaming), a thin adapter process is trivial to write and maintain independently.

**Third-party adapters can be higher quality.** A Discord bot adapter maintained by someone who deeply knows Discord's API quirks will be better than one built by the core Pekobot team. The community is better positioned to build and maintain these.

**Scope discipline is a design principle.** This decision is consistent with the broader principle that the core runtime should do its job (run agents) and not accumulate peripheral features. Channels are peripheral.

## Tradeoffs accepted

- Users who want Discord or Slack integration must find or build an adapter. For early adopters, this is friction.
- The webhook approach requires a publicly accessible URL (or tunnel) for inbound webhooks. This adds setup complexity compared to a long-polling integration that would work behind NAT.
- Documentation must explain this clearly to avoid users expecting built-in Discord support.

## Alternatives considered

**First-party integrations as optional plugins.** Rejected. Still requires the core team to maintain them. Plugin status doesn't reduce maintenance burden.

**Community-contributed integrations in the main repo.** Rejected. Same maintenance concern. Better to have them in separate repositories with dedicated maintainers.

**First-party Discord only (most requested).** Rejected. Sets a precedent for adding more channels. Once Discord is in, Slack requests are difficult to decline.
