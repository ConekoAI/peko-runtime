# ADR-017: Unified Extension Architecture (Extensions 2.0)

**Date:** 2026-04-08
**Status:** Proposed
**Deciders:** Core team

## Context

Pekobot currently has multiple extension mechanisms that evolved independently:

| Extension Type | Hook Points | Storage | Lifecycle |
|---------------|-------------|---------|-----------|
| **Skills** | System prompt (`{{skills}}`) | `~/.pekobot/skills/` | Load on startup |
| **Universal Tools** | System prompt + Native tool calling | `~/.pekobot/tools/` | Discovery + registration |
| **MCP Servers** | System prompt + Native tool calling | `mcp_servers/` | Connection management |
| **Channels** | I/O streaming | Built-in / config | Per-session init |
| **Hooks** | Event bus, webhooks | `HookRegistry` | Register/unregister |
| **Gateways** | Message I/O | `~/.pekobot/gateways/` | Plugin loading |

Each system has its own:
- Discovery mechanism
- Configuration format
- Registration API
- Lifecycle management
- Storage location

This creates several problems:

**1. Inconsistent User Experience**
- Users learn different commands for each extension type
- Installation/uninstallation follows different patterns
- No unified way to list, enable, or disable extensions

**2. Code Duplication**
- Each extension type implements its own file watching
- Each has its own manifest parsing
- Each manages its own enable/disable state

**3. Cross-Cutting Concerns are Hard**
- Telemetry must instrument each system separately
- Permission checks scattered across modules
- No unified dependency resolution

**4. Composability is Limited**
- A Discord gateway cannot easily provide both I/O and tools
- An extension cannot hook into multiple lifecycle points cleanly
- No mechanism for extensions to depend on each other

## Decision

Implement a **Unified Extension Architecture** with three layers:

1. **Extension Core**: Central registry of all hook points in the agentic loop
2. **Extension Type Adapters**: Map specific extension formats to hook points
3. **Extension Manager**: Unified lifecycle management (install/enable/disable/uninstall/bundle)

### Hook Points

All hook points in the agentic loop are explicitly defined:

```rust
pub enum HookPoint {
    // Prompt lifecycle
    PromptSystemSection { section: String, priority: i32 },
    PromptPreProcess,
    PromptPostProcess,
    
    // Tool lifecycle
    ToolRegister,
    ToolExecute { tool_name: String },
    ToolResultTransform,
    
    // Session lifecycle
    SessionStateChange,
    SessionCompaction,
    SessionContextBuild,
    
    // I/O lifecycle
    ChannelInput,
    ChannelOutput,
    MessagePreSend,
    MessagePostReceive,
    
    // Event lifecycle
    EventSubscribe { topic_pattern: String },
    EventEmit,
    
    // Agent lifecycle
    AgentInit,
    AgentShutdown,
    AgentIteration { iteration: usize },
}
```

### Adapter Pattern

Each extension type implements the `ExtensionTypeAdapter` trait:

```rust
pub trait ExtensionTypeAdapter: Send + Sync {
    fn extension_type(&self) -> &'static str;
    fn manifest_format(&self) -> ManifestFormat;
    fn resolve_hooks(&self, manifest: &ExtensionManifest) -> Vec<HookBinding>;
    fn lifecycle_manager(&self) -> Box<dyn LifecycleManager>;
}
```

**Skill Adapter Example:**
- Detects: `SKILL.md` files with YAML frontmatter
- Hooks: `PromptSystemSection { section: "skills" }` only
- Lifecycle: Stateless (no initialization needed)

**MCP Adapter Example:**
- Detects: `config.json` with MCP schema
- Hooks: `ToolRegister`, `PromptSystemSection`, `ToolExecute`
- Lifecycle: Stateful (starts server process, manages connection)

### Unified Commands

```bash
# Install any extension type (auto-detected)
pekobot ext install ./my-skill
pekobot ext install ./mcp-server.json
pekobot ext install ./discord-gateway

# List all extensions regardless of type
pekobot ext list
# ID           TYPE      STATUS   HOOKS
# docker       skill     enabled  prompt:skills
# filesystem   mcp       enabled  prompt:tools, tool:register, tool:execute
# discord      gateway   enabled  channel:input, channel:output, event:emit

# Enable/disable
pekobot ext disable docker
pekobot ext enable docker

# Bundling
pekobot ext bundle create production-bundle \
    --with docker \
    --with deploy-skill \
    --with k8s-mcp
pekobot ext bundle install ./production-bundle.tar.gz
```

## Reasoning

**Single Mental Model.** Users learn one set of commands for all extensions. The `pekobot ext` namespace becomes the universal interface for extending agent capabilities.

**DRY Principle.** File watching, manifest parsing, enable/disable state, and telemetry are implemented once in the Extension Manager, not duplicated across skills, MCP, channels, etc.

**Composability.** Extensions can hook into multiple points naturally. A Discord gateway can provide:
- `ChannelInput`/`ChannelOutput` for I/O
- `ToolRegister` for platform-specific tools (e.g., "send_discord_message")
- `EventEmit` for Discord-specific events (mentions, reactions)

**Cross-Cutting Concerns.** Telemetry, permissions, and dependency resolution work uniformly:
```rust
// One place to add logging
impl HookHandler for LoggingWrapper {
    async fn handle(&self, ctx: HookContext) -> HookResult {
        let start = Instant::now();
        let result = self.inner.handle(ctx).await;
        telemetry.record_hook_duration(ctx.point, start.elapsed());
        result
    }
}
```

**Custom Extension Types.** Users can define custom adapters for internal needs:
```rust
// Company-specific extension format
pub struct InternalToolAdapter;
impl ExtensionTypeAdapter for InternalToolAdapter {
    fn extension_type(&self) -> &'static str { "internal-tool" }
    fn resolve_hooks(&self, manifest: &ExtensionManifest) -> Vec<HookBinding> {
        vec![HookBinding::tool_register_with_auth(manifest)]
    }
}
extension_manager.register_adapter(Box::new(InternalToolAdapter));
```

**Progressive Disclosure.** Simple extensions (skills) only use one hook. Complex extensions (gateways) can use many. The system scales with complexity.

## Tradeoffs Accepted

**Migration Cost.** Existing skills, MCP servers, and tools must be migrated to the new system. See migration plan below for phased approach.

**Performance Overhead.** A layer of indirection through the Extension Core adds minor overhead per hook invocation. Mitigated by:
- Handlers are cached after resolution
- Disabled extensions are skipped entirely
- Priority ordering prevents unnecessary invocations

**Complexity in HookPoint Definition.** The enum must capture all possible hook points. Adding new hook points requires core changes. Mitigated by:
- Generic hook points with typed payloads (e.g., `ToolExecute { tool_name: String }`)
- Escape hatch: extensions can register raw callbacks for edge cases

**Ordering Dependencies.** Multiple handlers at the same hook point may depend on execution order. Addressed by:
- Explicit priority values (i32, higher = earlier)
- Bundle manifests can declare ordering constraints

## Alternatives Considered

**Keep Current System.** Rejected. Duplication, inconsistent UX, and composability problems compound as more extension types are added.

**Plugin-based (Dynamic Loading).** Rejected. Dynamic libraries add deployment complexity and security concerns. The adapter pattern achieves similar flexibility with static linking.

**Microkernel Architecture.** Rejected. Running extensions as separate processes (like MCP) is overkill for stateless extensions like skills. The hybrid approach (stateless in-process, stateful out-of-process) provides better performance where needed.
