# ADR-007: Built-in tools compiled into runtime, not loaded as plugins

**Date:** 2026-03-16
**Status:** Accepted
**Deciders:** Core team

## Context

The 13 built-in tools (`filesystem`, `process`, `apply_patch`, etc.) could either be compiled directly into the runtime binary or loaded at startup as plugins (shared libraries or scripts). The plugin approach would make the built-ins more easily replaceable and independently updatable.

## Decision

All 13 built-in tools are compiled into the runtime binary. They are not plugins, not scripts, and not separately loadable. They can be disabled per-agent via `disabled_tools` but not replaced by a user-provided implementation with the same name (the built-in always wins in name conflicts; a warning is logged).

## Reasoning

**Security-critical tools must not be replaceable.** The sandbox enforcement in `filesystem` and `process` (path canonicalisation, shell blocking, credential stripping) is a security boundary. If these tools could be replaced by user-provided scripts, an agent image could ship a malicious `filesystem` tool that bypasses the sandbox. Compiling them in makes the enforcement unconditional.

**Startup reliability.** A runtime that depends on externally-loaded tools for basic operations (`filesystem`, `session_status`) has more failure modes than one that doesn't. A missing plugin means the tool is unavailable; a compiled-in tool is always there.

**Performance.** Built-in tools are Rust code in the same process. There is no serialisation, no subprocess spawn, no stdin/stdout round-trip. For tools called hundreds of times per session (`filesystem.read`, `session_status`), this matters.

**Single binary deployment.** One of the core design goals is that `pekobot` is a single binary that can be copied to any machine and run. Compiled-in tools preserve this. Plugin-loaded tools would require a `lib/` directory alongside the binary.

**The custom tool system handles extensibility.** Users who need different behaviour can disable a built-in and provide their own tool with a different name, or use an MCP server. The extension points are rich; the built-ins just don't participate in them.

## Tradeoffs accepted

- Built-ins cannot be updated without updating the runtime binary. If a built-in has a bug, users must upgrade `pekobot`.
- Adding a new built-in requires a runtime release. We commit to keeping the built-in set stable and small.
- Users cannot override a built-in with a same-named custom tool (built-in always wins). This is intentional for security but occasionally inconvenient.

## Alternatives considered

**All tools as plugins (scripts or shared libraries).** Rejected. Security boundary violation, startup reliability risk, performance overhead.

**Built-ins as scripts bundled in the binary (e.g. embedded Python).** Rejected. Embedding an interpreter adds significant binary size and startup overhead. Rust is the correct tool for security-critical code that must run in the same process.

**Optional: allow replacing built-ins via explicit capability grant.** Deferred. Could be reconsidered in Phase 2 under control plane policy enforcement, but only with signing and verification of the replacement.
