# ADR-013: Stateless Agent Runtime (Cold-Start Model)

**Date:** 2026-03-18
**Status:** Accepted
**Deciders:** Core team

## Context

The `pekobot` architecture originally assumed agents would have a lifecycle (Starting → Running → Stopping → Stopped) with the daemon maintaining agent state in RAM. This led to complexity around:
- Instance state management
- Warm vs cold start tradeoffs
- RAM usage when agents are "running" but idle
- Offline vs online command differentiation

Upon analysis, we realized **nothing actually needs to stay resident in RAM** between requests. All agent state is either:
- Configuration (TOML files on disk)
- Session history (JSONL files on disk)
- Provider clients (lightweight, can be recreated)
- Tools (inexpensive to instantiate)

The only persistent services are external (MCP servers, vector stores), which are stateless connections.

## Decision

**Agents are stateless and cold-start on every request.**

The daemon does not keep agents "warm" in RAM. When a request arrives (via API or cron trigger), the daemon (or CLI) performs:

1. Load agent config from disk
2. Resolve active session (from `.active.json` or create new)
3. Load session history from JSONL
4. Instantiate tools and provider
5. Run agentic loop
6. **Exit and free all resources**

This applies to:
- API requests (`POST /agents/{id}/chat`)
- Cron job triggers
- Webhook triggers
- CLI commands (`agent start -m`, `session send`)

### Unified Message Command

Both `agent start -m` and `session send` have been unified into a single top-level command:

```bash
# Primary: Send a message to an agent (cold-start, no daemon required)
pekobot send <agent> <message>
pekobot send my-agent "Hello"

# With options
pekobot send my-agent "Hello" --session sess_xxx    # Specific session
pekobot send my-agent "Hello" --new                  # Force new session
pekobot send my-agent --file prompt.txt              # Read from file
echo "Hello" | pekobot send my-agent --stdin         # Read from stdin
```

Both CLI and API paths execute the same cold-start sequence. The daemon only provides:
- HTTP API endpoint for external integrations
- Cron scheduling (triggers spawn tasks that cold-start)
- Webhook reception

#### Deprecation Notice

The following commands are **deprecated** and will be removed in a future version:
- `pekobot agent start <agent> --message "..."` → Use `pekobot send <agent> "..."`
- `pekobot agent start <agent> -M "..."` → Use `pekobot send <agent> "..."`
- `pekobot session send <instance> "..."` → Use `pekobot send <agent> "..."`

Deprecated commands will show a warning pointing to the new `send` command.

## Reasoning

**Cold-start latency is acceptable.**

| Operation | Time |
|-----------|------|
| Load config TOML | < 10ms |
| Load session JSONL (< 1000 events) | < 50ms |
| Instantiate built-in tools | < 20ms |
| Connect to provider | < 100ms |
| **Total cold-start** | **< 200ms** |

Compared to LLM response times (1-10s), this is negligible. MCP server spawning (1-10s) happens regardless of warm/cold state.

**Zero RAM at idle.**

No agent consumes memory when not processing a request. The daemon itself is lightweight (SQLite, HTTP router, cron scheduler).

**Simplified mental model.**

Users don't need to think about "starting" or "stopping" agents. Agents exist as configuration; they "run" only while processing a request, then immediately exit.

**Cron reliability without complexity.**

Cron jobs work because the daemon is always-on (like Ollama). On trigger, it spawns a task that cold-starts the agent. No need for agents to stay "warm" just for cron.

**Filesystem is source of truth.**

No state synchronization issues. The SQLite `state.db` tracks only:
- Instance metadata (image ref, created_at)
- Cron job schedules
- Active session hints (for API routing)

All can be rebuilt from filesystem if deleted.

## Consequences

### For Users

- **No "instance lifecycle" to manage** - agents don't need to be "started"
- **Consistent behavior** - CLI and API both cold-start
- **Lower resource usage** - zero RAM when idle
- **Slightly slower first response** - but dominated by LLM latency anyway

### For Implementation

- **Remove InstanceState enum** - no Running/Stopped states
- **Simplify daemon** - just API routing + task spawning
- **Unified execution path** - CLI and API share same cold-start code
- **MCP cold-start per request** - acceptable tradeoff for simplicity

### Architecture Changes

**Removed:**
- Instance lifecycle state machine
- Warm/cold instance distinction
- Instance memory management
- Complex idle/hibernate logic

**Retained:**
- Daemon always-on (for API, cron, webhooks)
- JSONL session storage
- SQLite index (lightweight, rebuildable)

## Implementation Notes

### Request Flow

```
Request arrives (HTTP / Cron / CLI)
    │
    ▼
┌─────────────────┐
│  Load Config    │  ← Read TOML from disk
│  (cold start)   │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌───────┐ ┌──────────┐
│  API  │ │  Direct  │
│ Route │ │   Mode   │
└───┬───┘ └────┬─────┘
    │          │
    └────┬─────┘
         ▼
┌─────────────────┐
│ Load Session    │  ← Read .jsonl from disk
│ (history)       │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Instantiate     │  ← Create tools, provider client
│ Runtime         │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Run Agentic     │  ← LLM calls, tool execution
│ Loop            │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Exit & Cleanup  │  ← Free all RAM
└─────────────────┘
```

### Cron Execution

```rust
// Daemon schedules cron job
// On trigger:
tokio::spawn(async move {
    // Cold-start agent
    let agent = Agent::from_config(config_path).await?;
    // Run with trigger payload
    agent.run_once(trigger_message).await?;
    // Exit - all resources freed
    Ok(())
});
```

### MCP Handling

MCP servers are spawned per-request. This is slower (1-10s) but:
- Ensures fresh state
- No orphaned processes
- Matches serverless model

For frequently-used MCPs, users can run them externally and reference via config.

## Alternatives Considered

**Warm instances (original design).** Rejected. Adds complexity (lifecycle management, idle detection, memory pressure) for minimal benefit (200ms saved). Most latency is LLM anyway.

**Hybrid warm/cold.** Rejected. Unpredictable performance. Users can't reason about when cold-start happens.

**Daemon-less cron.** Rejected. Requires OS-level cron integration. Less portable, harder to manage.
