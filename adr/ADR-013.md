# ADR-013: Offline-First Session Commands

**Date:** 2026-03-18
**Status:** Accepted
**Deciders:** Core team

## Context

The `pekobot session` CLI commands currently require the daemon to be running for most operations. This creates friction for users who want to:

- List sessions while the daemon is stopped
- View session history without starting the daemon
- Branch or delete sessions without daemon coordination
- Manage sessions on remote/headless systems where starting the daemon is inconvenient

The session storage architecture (JSONL files + sidecar `.index.json` files) was designed to support human-readable, git-friendly storage. This same architecture enables offline CLI operations.

## Decision

All session commands that operate on stored data (list, show, branch, delete) are implemented as **offline-first** operations that read/write directly from the filesystem. Only commands that inherently require a running agent (send, switch on active instances) require daemon coordination.

The `active_session_id` concept is split:
- **Runtime active session**: Stored in daemon's SQLite `state.db`, used when agent is running
- **Preferred session hint**: Stored in `<workspace>/sessions/.active.json`, set by CLI, read by daemon on startup

## Reasoning

**The filesystem is the source of truth.** Session data is already stored as JSONL files with sidecar indices. The daemon's SQLite database is explicitly documented as a "read-optimised index" that can be rebuilt from the filesystem (ADR-001). CLI commands should respect this architecture and work directly with the source of truth.

**Offline operations are a common case.** Users frequently inspect session history, clean up old sessions, or branch from previous conversations without needing the agent to run. Requiring daemon startup for these read-only or file-only operations is unnecessary friction.

**No coordination needed for most operations.** Branching a session is a file copy. Deleting is a file removal. Listing is reading directory contents. None of these require coordination with a running process or access to shared mutable state.

**Active session is a runtime concern.** The concept of "which session is currently active" only matters when an agent instance is running and receiving messages. When the daemon is stopped, "switching active session" is merely setting a preference for the next startup. This can be persisted to a simple file.

**CLI responsiveness matters.** Starting the daemon (even in foreground) adds seconds of latency to simple commands. Offline commands are instant, matching user expectations for local file operations.

## Implementation Details

### Offline-Capable Commands

| Command | Operation | Filesystem Action |
|---------|-----------|-------------------|
| `session list` | Read metadata | Scan `*.index.json` in sessions dir |
| `session show` | Read history | Parse `{session_id}.jsonl` |
| `session branch` | Copy + create | Copy JSONL, create sidecar with `parent_session_id` |
| `session delete` | Delete files | Remove `.jsonl` and `.index.json` |
| `session switch` | Set preference | Write `.active.json` with chosen session ID |

### Daemon-Required Commands

| Command | Why Daemon Required |
|---------|---------------------|
| `session send` | Needs running agent to process message |
| `session switch` (when running) | Must notify running instance of session change |

### File Locations

Sessions are stored in priority order:
1. `{data_dir}/agents/{instance_id}/sessions/` (runtime instances)
2. `{config_dir}/teams/{team}/agents/{agent_name}/sessions/` (team agents)
3. `{config_dir}/agents/{agent_name}/sessions/` (standalone agents)

The `.active.json` file (CLI-managed preference):
```json
{
  "session_id": "sess_xxx",
  "set_at": "2026-03-18T12:00:00Z",
  "set_by": "cli"
}
```

## Tradeoffs Accepted

- **No atomicity guarantees.** If a crash happens mid-branch (after JSONL copy but before sidecar creation), the session may be in an inconsistent state. Mitigated by idempotent sidecar creation on daemon startup.

- **Multiple writers possible.** Both CLI and daemon can write to the sessions directory. File locking (via `SessionStorage` atomic writes) prevents corruption of individual files, but there's no global coordination. This is acceptable because:
  - Sessions are append-only (no concurrent modifications to same file)
  - New sessions get unique IDs (UUID-based)
  - Deletion is idempotent

- **Active session ambiguity.** If CLI sets `.active.json` while daemon is running, the daemon won't notice until restart. This is acceptable because:
  - The CLI warns when daemon is running
  - Users typically don't switch sessions while actively chatting
  - Restarting the agent picks up the new preference

- **Cross-platform path handling.** The CLI must locate sessions across different directory structures (data vs config, team vs standalone). This adds complexity but is manageable with a lookup priority system.

## Alternatives Considered

**Daemon-only operations.** Rejected. Forces unnecessary daemon startup for simple read/inspect operations. Violates the principle that the filesystem is the source of truth.

**Hybrid: Try daemon first, fall back to offline.** Rejected. Unpredictable behavior - success depends on daemon state. Users should explicitly know when they're working offline vs online.

**SQLite for CLI queries.** Rejected. SQLite is the daemon's implementation detail. CLI should work with the documented JSONL format, not depend on an internal index that may not exist or may be stale.

**REST API over Unix socket only.** Rejected. Socket requires daemon running. Doesn't solve the offline use case.
