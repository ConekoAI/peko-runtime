# ADR-016: Unified Session Resolution and Async Completion Architecture

**Date:** 2026-03-26
**Status:** Proposed
**Deciders:** Core team

## Context

The current implementation (post-ADR-015) has several architectural issues that impact performance, reliability, and maintainability:

### 1. Performance Issues

**Thread per request:** `StatelessAgentService::execute_streaming()` spawns a new thread with its own tokio runtime for every streaming request:

```rust
tokio::task::spawn_blocking(move || {
    let rt = tokio::runtime::Builder::new_current_thread()
        .enable_all()
        .build()
        .expect("Failed to create local runtime");
    rt.block_on(async { /* execution */ })
});
```

This creates significant overhead:
- Thread creation cost
- Runtime initialization cost
- Double runtime (main + spawned) memory overhead
- No thread pool reuse

**Artificial delays:** The 500ms grace period after `End` event adds latency to every request:

```rust
LifecyclePhase::End => {
    tokio::time::sleep(Duration::from_millis(500)).await; // Wasteful
    break;
}
```

**Extra abstraction layer:** `MessageService` adds an unnecessary indirection that:
- Duplicates session resolution logic
- Adds an extra `tokio::spawn` for event forwarding in streaming mode
- Increases code complexity without adding value

### 2. Reliability Issues

**Race condition in session persistence:** The spawned task may be dropped before file I/O completes:

```rust
tokio::spawn(async move {
    let _ = handle.await; // Fire and forget
});
```

When the main thread receives the `End` event and exits, the spawned task may be mid-write, causing session corruption or missing entries.

**Non-deterministic completion:** There's no mechanism to ensure:
- Session write operations complete before `End` event
- File buffers are flushed to disk
- Task cleanup finishes properly

### 3. Architectural Violations

**SRP Violation:** `StatelessAgentService` has too many responsibilities:
- Session resolution (should be `SessionManager`'s job)
- Agent execution
- Streaming orchestration
- History loading
- Prompt building

**DRY Violation:** Session resolution logic is duplicated:
- `StatelessAgentService::execute()` - uses direct `SessionManager::for_cli()` + `open_session()` pattern
- `StatelessAgentService::execute_streaming()` - copies the same logic
- `MessageService::send_message()` - uses deprecated `SessionResolver`
- `MessageService::send_message_unified()` - uses `SessionResolver`

**Inconsistent Abstractions:** Three different session resolution patterns:
1. Direct `SessionManager::for_cli()` + `open_session()` - used by `StatelessAgentService`
2. `SessionResolver::resolve_session()` - deprecated, used by `MessageService`
3. `SessionManager::resolve_session()` - the actual single authority, not used consistently

### 4. Current Workaround Issues

The 500ms grace period (hotfix for session logging) is:
- **Wasteful:** Always waits 500ms even if I/O completes in 10ms
- **Unreliable:** File I/O might take longer than 500ms on slow disks
- **Poor UX:** 50% latency increase for 1-second responses

## Decision

**Refactor to use `SessionManager` as the single authority for session resolution, add async completion signals to eliminate race conditions, merge `MessageService` into `StatelessAgentService` to remove unnecessary abstraction, and remove the artificial grace period.**

### Core Principles

1. **Single Runtime:** All execution happens on the main tokio runtime
2. **Explicit Completion:** Tasks signal completion through channels, not timeouts
3. **Single Authority:** `SessionManager` is the sole source of truth for session operations
4. **Clear Boundaries:** Enforce SRP - `SessionManager` manages sessions, `StatelessAgentService` executes agents
5. **Flat Architecture:** Remove unnecessary abstraction layers
6. **Provider Callback/Hooks:** Use async completion callbacks instead of polling

### Key Components

#### 1. SessionManager as Single Authority

`SessionManager` already has `resolve_session()` - the correct single authority. No additional serialization mechanism is needed because:

1. **Session writes are already synchronous** - `SessionStorage::append_event()` uses `atomic_append()` which blocks until the write completes
2. **FileLock already serializes** concurrent access to the same session file
3. **The race condition is timing, not concurrency** - it's about ensuring the consumer waits for the write to complete, not about serializing multiple writers

```rust
impl SessionManager {
    /// Existing: resolve session with strategy (already the authority)
    pub async fn resolve_session(
        &mut self,
        agent_name: &str,
        team: Option<&str>,
        channel: ChannelType,
        channel_id: &str,
        session_id: Option<String>,
        force_new: bool,
    ) -> Result<ResolvedSession> {
        // Already the single source of truth - KEEP THIS
    }
}
```

#### 2. Async Completion Signals (No Artificial Delays)

The core fix is adding completion signals to `EventStream`:

```rust
pub struct EventStream {
    pub receiver: mpsc::Receiver<AgenticEvent>,
    pub completion: oneshot::Receiver<Result<()>>, // NEW
    pub session_id: String,
    pub is_new_session: bool,
}
```

**Producer side** (`StatelessAgentService::execute_streaming`):

```rust
pub async fn execute_streaming(
    &self,
    request: ExecutionRequest,
) -> Result<EventStream> {
    let (event_tx, event_rx) = mpsc::channel(1000);
    let (completion_tx, completion_rx) = oneshot::channel();
    
    // Single runtime - no spawn_blocking
    tokio::spawn(async move {
        // Execute and collect result
        let result = self.execute_with_session(request, session, |event| {
            let _ = event_tx.send(event);
        }).await;
        
        // At this point, all session writes have completed because
        // add_assistant/add_tool_result are synchronous
        
        // Signal completion
        let _ = completion_tx.send(result.map(|_| ()));
    });
    
    Ok(EventStream {
        receiver: event_rx,
        completion: completion_rx,
        session_id: request.session_id,
    })
}
```

**Consumer side** (channels await completion):

```rust
impl EventStream {
    /// Process events with proper completion tracking
    pub async fn process<C: Channel>(
        self,
        channel: &C,
    ) -> Result<ChannelOutput> {
        let mut output = ChannelOutput::new(&self.session_id);
        
        // Process all events from receiver
        while let Some(event) = self.receiver.recv().await {
            match &event {
                AgenticEvent::Lifecycle { phase: End, .. } => {
                    // Process the event but don't break yet
                }
                _ => {}
            }
            channel.process_event(event).await?;
        }
        
        // Receiver closed - NOW wait for completion signal
        // This ensures session persistence is complete
        match tokio::time::timeout(
            Duration::from_secs(30),
            self.completion
        ).await {
            Ok(Ok(())) => Ok(output),
            Ok(Err(e)) => Err(e),
            Err(_) => Err(anyhow!("Completion timeout")),
        }
    }
}
```

#### 3. Merged StatelessAgentService (Replaces MessageService)

**Before:** Three layers (API → MessageService → StatelessAgentService)

**After:** Two layers (API → StatelessAgentService)

```rust
//! Stateless Agent Service - Cold-start execution with request/response handling
//! 
//! This module replaces the separate MessageService by integrating
//! request/response handling directly into the execution service.

// Request/Response types (moved from MessageService)
pub struct MessageRequest {
    pub agent_name: String,
    pub team: Option<String>,
    pub message: String,
    pub session_id: Option<String>,
    pub new_session: bool,
    pub timeout_secs: Option<u64>,
}

pub struct MessageResult {
    pub content: String,
    pub session_id: String,
    pub is_new_session: bool,
    pub usage: TokenUsage,
    pub tool_calls: Vec<ToolCallInfo>,
    pub duration_ms: u64,
    pub iterations: usize,
    pub success: bool,
    pub error: Option<String>,
}

pub struct StatelessAgentService {
    config_service: Arc<AgentConfigService>,
    path_resolver: PathResolver,
}

impl StatelessAgentService {
    /// High-level: Execute message and return blocking response
    /// 
    /// Replaces MessageService::send_message()
    pub async fn execute_message(&self, request: MessageRequest) -> Result<MessageResult> {
        let start = Instant::now();
        
        // Resolve session using SessionManager (single authority)
        let mut session_manager = SessionManager::for_cli(
            self.path_resolver.clone(),
            &request.agent_name,
            self.get_team(&request.agent_name).await?,
        );
        
        let resolved = session_manager
            .resolve_session(
                &request.agent_name,
                request.team.as_deref(),
                ChannelType::Http,
                "default",
                request.session_id.clone(),
                request.new_session,
            )
            .await?;
        
        // Build and execute
        let exec_request = ExecutionRequest {
            agent_name: request.agent_name.clone(),
            session_id: resolved.session_id.clone(),
            message: request.message,
            context: None,
            timeout_secs: request.timeout_secs,
        };
        
        let result = self.execute_with_session(
            exec_request, 
            resolved.context.hybrid.base.clone()
        ).await;
        
        // Transform to MessageResult
        let duration_ms = start.elapsed().as_millis() as u64;
        match result {
            Ok(exec_result) => Ok(MessageResult {
                content: exec_result.response,
                session_id: resolved.session_id,
                is_new_session: resolved.is_new,
                usage: exec_result.usage,
                tool_calls: exec_result.tool_calls.into_iter().map(|tc| ToolCallInfo {
                    id: format!("tool_{}", Uuid::new_v4().simple()),
                    name: tc.name,
                    parameters: tc.parameters,
                    result: tc.result,
                }).collect(),
                duration_ms,
                iterations: exec_result.iterations,
                success: exec_result.success,
                error: exec_result.error,
            }),
            Err(e) => Err(e),
        }
    }
    
    /// High-level: Execute message and return EventStream for streaming
    /// 
    /// Replaces MessageService::send_message_unified()
    /// 
    /// NO EXTRA SPAWN - directly returns EventStream from execute_streaming
    pub async fn execute_message_streaming(
        &self,
        request: MessageRequest,
    ) -> Result<EventStream> {
        // Resolve session
        let mut session_manager = SessionManager::for_cli(
            self.path_resolver.clone(),
            &request.agent_name,
            self.get_team(&request.agent_name).await?,
        );
        
        let resolved = session_manager
            .resolve_session(
                &request.agent_name,
                request.team.as_deref(),
                ChannelType::Http,
                "default",
                request.session_id.clone(),
                request.new_session,
            )
            .await?;
        
        // Build execution request
        let exec_request = ExecutionRequest {
            agent_name: request.agent_name.clone(),
            session_id: resolved.session_id.clone(),
            message: request.message,
            context: None,
            timeout_secs: request.timeout_secs,
        };
        
        // Direct call - NO extra spawn, NO extra channel hop
        let mut event_stream = self
            .execute_streaming_with_session(exec_request, resolved.context.hybrid.base.clone())
            .await?;
        
        // Attach metadata
        event_stream.session_id = resolved.session_id;
        event_stream.is_new_session = resolved.is_new;
        
        Ok(event_stream)
    }
    
    /// Low-level: Execute with raw ExecutionRequest (blocking)
    pub async fn execute(&self, request: ExecutionRequest) -> Result<ExecutionResult> {
        // Implementation delegates to execute_with_session
    }
    
    /// Low-level: Execute streaming with raw ExecutionRequest
    pub async fn execute_streaming(
        &self,
        request: ExecutionRequest,
    ) -> Result<EventStream> {
        // Implementation delegates to execute_streaming_with_session
    }
}
```

### Why NOT SessionActorQueue

After analyzing the actual session write flow, `SessionActorQueue` is **unnecessary** because:

**Current session write flow:**
1. Stream completes (all deltas accumulated in memory)
2. `add_assistant_with_blocks()` called - **synchronous write**
3. `SessionStorage::append_event()` → `atomic_append()` 
4. `atomic_append()` → `atomic_write()` → **file I/O with flush**
5. Returns only after write completes

**Already have serialization:**
- `FileLock` serializes concurrent access to the same session file
- Writes are synchronous (block until complete)
- No queue needed - writes don't return until persisted

**The actual race condition:**
- Consumer receives `End` event
- Consumer disconnects immediately
- But session write just completed (synchronous)
- **The timing issue is consumer disconnecting too early, not concurrent writes**

**Solution:** Completion signal is sufficient - no `SessionActorQueue` needed.

### Why NOT Create a New `SessionResolutionService`

After analysis, creating a new `SessionResolutionService` would:

1. **Violate DRY:** Duplicate `SessionManager::resolve_session()` logic
2. **Violate SRP:** Split session authority between two services
3. **Add confusion:** `SessionResolver` is already deprecated for this reason

**Current state analysis:**

| Component | Current Role | Issue |
|-----------|--------------|-------|
| `SessionManager` | Already has `resolve_session()` - single authority | Not used consistently |
| `SessionResolver` | Thin wrapper around `SessionManager` | Already deprecated |
| `StatelessAgentService` | Uses `for_cli()` + `open_session()` directly | Bypasses resolution strategy |

**Better approach:**
- Use `SessionManager::resolve_session()` directly (already the authority)
- Deprecate `SessionResolver` completely
- Refactor `StatelessAgentService` to delegate resolution

## Reasoning

### Why Provider Callback/Hooks Approach?

**Callbacks are more explicit than polling:**
- Polling (current): `sleep(500ms)` and hope it's done
- Callbacks: Task explicitly signals "I am done with all work"

### Why is the Grace Period Completely Removed?

With the old architecture (spawn_blocking), we needed the grace period because:
1. Thread A spawned with `spawn_blocking` runs independently
2. Thread A could be writing to session file when main thread receives `End`
3. Main thread exits, Thread A gets killed mid-write → corrupted session

With ADR-016's single-runtime + completion signals:
1. Single `tokio::spawn` task runs on main runtime (not separate thread)
2. Task signals completion **only after** all session writes complete
3. Session writes are synchronous - they block until file I/O completes
4. Channel holds the task alive until completion is awaited
5. No fire-and-forget: the task is owned by the completion receiver

**Result:** The 500ms grace period is completely unnecessary. The completion signal is deterministic.

**Async-native:**
- Rust async is designed around futures and completion
- Callbacks integrate naturally with `oneshot::channel`
- No blocking operations needed

**Deterministic:**
- Order of operations is explicit
- Session write completes → Return from write method → Signal completion → End event processed → Await completion
- No guessing about timing

### Why Single Runtime?

**Resource efficiency:**
- One thread pool for all tasks
- No runtime creation overhead per request
- Better memory utilization

**Simpler mental model:**
- All async code runs on same runtime
- No `spawn_blocking` + `block_on` nesting
- Easier to debug and profile

**Better error handling:**
- Errors propagate naturally through async boundary
- No cross-runtime error conversion

### Why Flat Architecture?

**Before (MessageService layer):**
```
API → MessageService → StatelessAgentService → Engine
  │         │                    │
  │         └─ spawns task       └─ spawns task (spawn_blocking)
  └─ calls
```

**After (direct):**
```
API → StatelessAgentService → Engine
  │              │
  │              └─ single spawn (tokio::spawn)
  └─ calls
```

**Benefits:**
- Less indirection
- Easier to trace execution
- Fewer edge cases
- Simpler testing

### Why Keep `SessionManager` as Single Authority?

**Already the single source of truth:**
- `SessionManager::resolve_session()` already implements the resolution strategy
- Creating a new service would duplicate this logic (DRY violation)
- All session operations belong in one place

**Testability:**
- Mock `SessionManager` for tests
- Isolated unit testing
- No integration test needed for resolution logic

## Implementation Strategy

### Phase 1: StatelessAgentService Enhancement (Foundation)

1. Add `MessageRequest` and `MessageResult` types to `stateless_service.rs`
2. Add `execute_message()` method (blocking)
3. Add `execute_message_streaming()` method (streaming with EventStream)
4. Implement `get_team()` helper to avoid repetition

**Risk:** Low - internal refactoring, no external behavior change

### Phase 2: Completion Signals (Reliability)

1. Add `completion` channel to `EventStream`
2. Update `StatelessAgentService::execute_streaming()` to send completion signal
3. Update `event_stream_to_sse` to await completion
4. Update `default_process_stream` to await completion
5. Update CLI channel to await completion (remove 500ms sleep)
6. Remove 500ms grace period workaround

**Risk:** Medium - changes timing semantics, thorough testing needed

### Phase 3: Single Runtime (Performance)

1. Remove `spawn_blocking` from `execute_streaming`
2. Use `tokio::spawn` on main runtime
3. Add stress tests for concurrent execution

**Risk:** Medium - may expose race conditions previously hidden by thread isolation

### Phase 4: Merge MessageService (Cleanup)

1. Update API routes to call `StatelessAgentService` directly
2. Update CLI commands to call `StatelessAgentService` directly
3. Remove `MessageService` and `SessionResolver` completely
4. Update `AppState` to remove `message_service()` method
5. Update documentation

**Risk:** Low - cleanup only, all functionality preserved

## Alternatives Considered

### Alternative 1: Create New `SessionResolutionService`

Create a new service that wraps `SessionManager::resolve_session()`.

**Rejected:**
- Would duplicate existing `SessionManager` functionality (DRY violation)
- Would split session authority between two services (SRP violation)
- `SessionResolver` already exists and is deprecated for this reason

### Alternative 2: SessionActorQueue for Serialization

Add per-session actor queue for write serialization.

**Rejected:**
- Session writes are already synchronous (blocking until complete)
- `FileLock` already serializes concurrent access
- The race condition is timing (consumer disconnects too early), not concurrent writes
- Completion signal is sufficient and much simpler

### Alternative 3: Keep MessageService Separate

Keep the MessageService abstraction layer.

**Rejected:**
- Adds no value - just forwards calls
- Duplicates session resolution
- Adds extra spawn and channel hop
- Increases complexity without benefit

### Alternative 4: Keep Current Architecture, Improve Thread Pool

Use `rayon` or custom thread pool instead of `spawn_blocking` per request.

**Rejected:** Doesn't solve the fundamental issues (artificial delays, race conditions, architectural violations).

## Implementation Notes

### Backward Compatibility

**External API:** No changes. HTTP API SSE endpoint remains identical.

**Internal API:** `EventStream` gains `completion` field:

```rust
// Before
pub struct EventStream {
    pub receiver: mpsc::Receiver<AgenticEvent>,
    pub session_id: String,
    pub is_new_session: bool,
}

// After
pub struct EventStream {
    pub receiver: mpsc::Receiver<AgenticEvent>,
    pub completion: oneshot::Receiver<Result<()>>, // NEW
    pub session_id: String,
    pub is_new_session: bool,
}
```

Channels that ignore `completion` continue to work (backward compatible), but should be updated to use it.

**Note on Grace Period:** The 500ms grace period is **completely removed** (not just reduced) because:
- Single runtime: task doesn't drop until completion is signaled
- Session writes are synchronous: they complete before the method returns
- Completion signal: sent only after all writes complete
- No fire-and-forget: channel holds the completion receiver until explicitly awaited
- Deterministic: completion signal = session persistence guaranteed

### Critical Implementation Details

#### 1. Session Writes Are Already Synchronous

**CRITICAL:** Do not add additional serialization. Session writes already block until complete:

```rust
// In AgenticLoopV4::run_streaming_loop
{
    let mut s = session.write().await;
    s.add_assistant_with_blocks(content_blocks, ...).await?;  // SYNC - blocks until written
}
// At this point, write is complete
```

#### 2. SSE Adapter Must Await Completion

The SSE adapter in `api/streaming.rs` MUST be updated to await completion:

```rust
LifecyclePhase::End => {
    // Send Done event to client
    let _ = sender.send(ChatSseEvent::Done { ... }).await;
    
    // Wait for completion signal before returning
    match tokio::time::timeout(Duration::from_secs(30), event_stream.completion).await {
        Ok(result) => return result,
        Err(_) => return Err(anyhow!("Completion timeout")),
    }
}
```

#### 3. No Sub-Threads Spawned

After ADR-016, the entire flow runs on the **main tokio runtime**:

```
┌────────────────────────────────────────────────────────────┐
│                  THREADING MODEL AFTER                      │
└────────────────────────────────────────────────────────────┘

Main Tokio Runtime
    │
    ├─ API Request Handler (async task)
    │   │
    │   ├─ StatelessAgentService::execute_message_streaming()
    │   │   │
    │   │   ├─ SessionManager::resolve_session() ─┐
    │   │   │                                     │ All on main runtime
    │   │   ├─ tokio::spawn(async {              │ NO OS threads spawned
    │   │   │   AgenticLoopV4::run_streaming()   │ NO spawn_blocking
    │   │   │})                                   │
    │   │   │                                     │
    │   │   └─ returns EventStream ───────────────┘
    │   │
    │   └─ event_stream.process(channel)
    │       └─ await completion signal
    │
    └─ (other concurrent tasks)

Single Runtime. No Thread per Request.
```

**Key points:**
- `tokio::spawn` creates async tasks, not OS threads
- All tasks share the main tokio thread pool
- `spawn_blocking` is completely removed
- Session writes are async I/O, not blocking
