# ADR-016: Unified Session Resolution and Async Completion Architecture

**Date:** 2026-03-26
**Status:** Proposed
**Deciders:** Core team

## Context

The current implementation (post-ADR-015) has several architectural issues that impact performance, reliability, and maintainability:

### 1. Performance Issues

**Thread per request:** `StatelessAgentService::execute_streaming()` spawns a new thread with its own tokio runtime for every streaming request:

```rust
tokio::task::spawn_blocking(move || {
    let rt = tokio::runtime::Builder::new_current_thread()
        .enable_all()
        .build()
        .expect("Failed to create local runtime");
    rt.block_on(async { /* execution */ })
});
```

This creates significant overhead:
- Thread creation cost
- Runtime initialization cost
- Double runtime (main + spawned) memory overhead
- No thread pool reuse

**Artificial delays:** The 500ms grace period after `End` event adds latency to every request:

```rust
LifecyclePhase::End => {
    tokio::time::sleep(Duration::from_millis(500)).await; // Wasteful
    break;
}
```

### 2. Reliability Issues

**Race condition in session persistence:** The spawned task may be dropped before file I/O completes:

```rust
tokio::spawn(async move {
    let _ = handle.await; // Fire and forget
});
```

When the main thread receives the `End` event and exits, the spawned task may be mid-write, causing session corruption or missing entries.

**Non-deterministic completion:** There's no mechanism to ensure:
- Session write operations complete before `End` event
- File buffers are flushed to disk
- Task cleanup finishes properly

### 3. Architectural Violations

**SRP Violation:** `StatelessAgentService` has too many responsibilities:
- Session resolution (should be `SessionManager`'s job)
- Agent execution
- Streaming orchestration
- History loading
- Prompt building

**DRY Violation:** Session resolution logic is duplicated:
- `StatelessAgentService::execute()` - one pattern
- `StatelessAgentService::execute_streaming()` - same logic copied
- `MessageService::send_message()` - uses `SessionResolver`
- `MessageService::send_message_unified()` - different pattern

**Inconsistent Abstractions:** Three different session resolution patterns:
1. Direct `SessionManager::for_cli()` + `open_session()`
2. `SessionResolver::resolve_session()`
3. `SessionManager::resolve_session()` (internal)

### 4. Current Workaround Issues

The 500ms grace period (hotfix for session logging) is:
- **Wasteful:** Always waits 500ms even if I/O completes in 10ms
- **Unreliable:** File I/O might take longer than 500ms on slow disks
- **Poor UX:** 50% latency increase for 1-second responses

## Decision

**Adopt a unified session resolution service with explicit async completion signals and single-runtime execution.**

### Core Principles

1. **Single Runtime:** All execution happens on the main tokio runtime
2. **Explicit Completion:** Tasks signal completion through channels, not timeouts
3. **Unified Resolution:** Single source of truth for session resolution
4. **Clear Boundaries:** Enforce SRP through dedicated services
5. **Provider Callback/Hooks:** Use async completion callbacks instead of polling

**Benefits:**
- One implementation used everywhere
- Consistent team-aware path resolution
- No duplication between CLI and API paths

#### 2. Async Completion Signals (No Artificial Delays)

Instead of spawning fire-and-forget tasks:

```rust
pub async fn execute_streaming(
    &self,
    request: ExecutionRequest,
) -> Result<EventStream> {
    let (event_tx, event_rx) = mpsc::channel(1000);
    let (completion_tx, completion_rx) = oneshot::channel();
    
    // Single runtime - no spawn_blocking
    tokio::spawn(async move {
        let result = self.execute_with_events(
            request,
            |event| { let _ = event_tx.send(event); },
        ).await;
        
        // Ensure all session writes complete BEFORE signaling completion
        if let Err(e) = &result {
            tracing::error!("Execution failed: {}", e);
        }
        
        // Signal actual completion
        let _ = completion_tx.send(result);
    });
    
    Ok(EventStream {
        receiver: event_rx,
        completion: completion_rx, // Caller can await this
        session_id: request.session_id,
    })
}
```

**Channel waits for actual completion:**

```rust
pub struct EventStream {
    pub receiver: mpsc::Receiver<AgenticEvent>,
    pub completion: oneshot::Receiver<Result<()>>,
    pub session_id: String,
}

impl EventStream {
    /// Process events with proper completion tracking
    pub async fn process<C: Channel>(
        self,
        channel: &C,
    ) -> Result<ChannelOutput> {
        let mut output = ChannelOutput::new(&self.session_id);
        
        // Process all events
        while let Some(event) = self.receiver.recv().await {
            match &event {
                AgenticEvent::Lifecycle { phase: End, .. } => {
                    // Don't break - wait for completion signal
                }
                _ => {}
            }
            channel.process_event(event).await?;
        }
        
        // NOW wait for actual completion (no artificial delay)
        match tokio::time::timeout(
            Duration::from_secs(30), // Reasonable timeout
            self.completion
        ).await {
            Ok(Ok(())) => Ok(output),
            Ok(Err(e)) => Err(e),
            Err(_) => Err(anyhow!("Completion timeout")),
        }
    }
}
```

**Benefits:**
- No artificial delays
- Deterministic completion
- Proper error propagation
- Single runtime (no thread spawning)

#### 3. Engine Layer with Completion Callbacks

```rust
impl AgenticLoopV4 {
    /// Execute with async completion callback
    pub async fn run_with_completion<F, C>(
        &self,
        prompt: &str,
        session: Arc<RwLock<UnifiedSession>>,
        history: Option<Vec<ChatMessage>>,
        on_event: F,
        on_complete: C,
    ) -> Result<AgenticResult>
    where
        F: Fn(AgenticEvent) + Send + Sync + 'static,
        C: FnOnce(Result<()>) + Send + 'static,
    {
        // Run the loop
        let result = self.run_loop(messages, session.clone(), &on_event, run_id).await;
        
        // Flush session before signaling completion
        {
            let s = session.read().await;
            s.storage.flush().await?;
        }
        
        // Signal completion
        on_complete(result.map(|_| ()));
        
        result
    }
}
```

**Benefits:**
- Session persistence guaranteed before completion
- No race conditions
- Explicit ordering of operations

#### 4. StatelessAgentService Refactoring (SRP Compliance)

```rust
pub struct StatelessAgentService {
    resolution_service: Arc<SessionResolutionService>,
    config_service: Arc<AgentConfigService>,
    path_resolver: PathResolver,
}

impl StatelessAgentService {
    /// Non-streaming execution
    pub async fn execute(&self, request: ExecutionRequest) -> Result<ExecutionResult> {
        // 1. Resolve session (delegated)
        let session = self.resolution_service
            .resolve(&request.agent_name, None, Some(&request.session_id), false)
            .await?;
        
        // 2. Execute (no session management)
        self.execute_with_session(request, session).await
    }
    
    /// Streaming execution - single runtime
    pub async fn execute_streaming(
        &self,
        request: ExecutionRequest,
    ) -> Result<EventStream> {
        // Same resolution path
        let session = self.resolution_service
            .resolve(&request.agent_name, None, Some(&request.session_id), false)
            .await?;
        
        // Single runtime execution with completion tracking
        self.create_event_stream(request, session).await
    }
    
    // Private: actual execution (DRY - single implementation)
    async fn execute_with_session(
        &self,
        request: ExecutionRequest,
        session: Arc<RwLock<UnifiedSession>>,
    ) -> Result<ExecutionResult> {
        // Execution logic only - no session resolution
    }
}
```

**Benefits:**
- Clear separation of concerns
- Single resolution path
- No code duplication
- Easier to test

## Reasoning

### Why Provider Callback/Hooks Approach?

**Callbacks are more explicit than polling:**
- Polling (current): `sleep(500ms)` and hope it's done
- Callbacks: Task explicitly signals "I am done with all work"

**Async-native:**
- Rust async is designed around futures and completion
- Callbacks integrate naturally with `oneshot::channel`
- No blocking operations needed

**Deterministic:**
- Order of operations is explicit
- Session write → Flush → Completion signal → End event
- No guessing about timing

### Why Single Runtime?

**Resource efficiency:**
- One thread pool for all tasks
- No runtime creation overhead per request
- Better memory utilization

**Simpler mental model:**
- All async code runs on same runtime
- No `spawn_blocking` + `block_on` nesting
- Easier to debug and profile

**Better error handling:**
- Errors propagate naturally through async boundary
- No cross-runtime error conversion

### Why Unified Session Resolution?

**Single source of truth:**
- All paths use same resolution logic
- Consistent behavior between CLI and API
- One place to fix bugs

**Testability:**
- Mock `SessionResolutionService` for tests
- Isolated unit testing
- No integration test needed for resolution logic

## Consequences

### Performance Improvements

| Metric | Before | After |
|--------|--------|-------|
| Thread creation per request | Yes | No |
| Runtime creation per request | Yes | No |
| Artificial delay | 500ms | 0ms |
| Memory per request | ~2MB (runtime) | ~few KB (task) |
| Latency overhead | 500ms+ | ~1-10ms |

### Reliability Improvements

| Aspect | Before | After |
|--------|--------|-------|
| Session persistence | Best-effort (race condition) | Guaranteed (completion signal) |
| Error propagation | May be lost in spawn | Preserved through channels |
| Determinism | Timing-dependent | Explicit ordering |

### Maintainability Improvements

| Aspect | Before | After |
|--------|--------|-------|
| Session resolution | 3 different patterns | 1 unified service |
| Code duplication | High (DRY violations) | Low (single implementations) |
| Testability | Hard (integration needed) | Easy (mock boundaries) |
| SRP compliance | Poor | Good |

## Implementation Strategy

### Phase 1: SessionResolutionService (Foundation)

1. Create `SessionResolutionService` with `resolve()` method
2. Migrate `MessageService` to use it
3. Migrate `StatelessAgentService` to use it
4. Deprecate old resolution paths

**Risk:** Low - internal refactoring, no external behavior change

### Phase 2: Completion Signals (Reliability)

1. Add `completion` channel to `EventStream`
2. Modify engine to signal completion after session flush
3. Update channels to await completion
4. Remove 500ms grace period workaround

**Risk:** Medium - changes timing semantics, thorough testing needed

### Phase 3: Single Runtime (Performance)

1. Remove `spawn_blocking` from `execute_streaming`
2. Use `tokio::spawn` with `LocalSet` if needed
3. Add stress tests for concurrent execution

**Risk:** Medium - may expose race conditions previously hidden by thread isolation

### Phase 4: Cleanup (Maintainability)

1. Remove deprecated resolution methods
2. Consolidate duplicate logic
3. Update documentation

**Risk:** Low - cleanup only

## Alternatives Considered

### Alternative 1: Keep Current Architecture, Improve Thread Pool

Use `rayon` or custom thread pool instead of `spawn_blocking` per request.

**Rejected:** Doesn't solve the fundamental issues (artificial delays, race conditions, architectural violations).

### Alternative 2: Blocking I/O with Synchronous Writes

Make session writes synchronous (blocking) within the spawned thread.

**Rejected:**
- Would block the thread for file I/O
- Doesn't solve race condition (still fire-and-forget)
- Against Rust async best practices

### Alternative 3: File System Notifications

Use OS file system notifications to detect when writes complete.

**Rejected:**
- Overly complex for the problem
- Platform-dependent
- Still requires waiting/polling

## Implementation Notes

### Backward Compatibility

**External API:** No changes. HTTP API SSE endpoint remains identical.

**Internal API:** `EventStream` gains `completion` field:

```rust
// Before
pub struct EventStream {
    pub receiver: mpsc::Receiver<AgenticEvent>,
    pub session_id: String,
}

// After
pub struct EventStream {
    pub receiver: mpsc::Receiver<AgenticEvent>,
    pub completion: oneshot::Receiver<Result<()>>, // NEW
    pub session_id: String,
}
```

Channels that ignore `completion` continue to work (backward compatible), but should be updated to use it.

### Testing Strategy

1. **Unit tests:** Mock `SessionResolutionService`, verify single resolution path
2. **Integration tests:** Verify completion signals work end-to-end
3. **Stress tests:** 100+ concurrent executions, verify no resource leaks
4. **Chaos tests:** Kill mid-execution, verify session consistency
