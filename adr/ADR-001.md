# ADR-001: JSONL for session storage, not SQLite

**Date:** 2026-03-15
**Status:** Accepted
**Deciders:** Core team

## Context

Sessions need to be durable, recoverable after crash, queryable by the API, and accessible to operators for debugging and audit. Two obvious options: store events in SQLite (structured, queryable, transactional) or as JSONL files (append-only, plain text, one file per session).

SQLite is already used in the project for the instance state index (`state.db`), so there was a natural pull toward putting everything there.

## Decision

Session events are stored as JSONL files. SQLite is used only as a read-optimised index over the JSONL files, not as the source of truth. If `state.db` is deleted or corrupted, it is rebuilt by scanning JSONL files on startup.

## Reasoning

**Human readability is a first-order requirement.** Sessions are the primary audit trail and the primary debugging tool. A developer investigating an agent's behaviour must be able to `cat` a session file and read it without tools. SQLite requires either a database client or a custom export command.

**Git-friendliness.** Agent workspaces are intended to be git repositories. JSONL files produce meaningful diffs (`git log`, `git diff`) that show exactly what happened in a session. A SQLite binary blob produces no useful diff.

**Append-only semantics are a natural fit.** Sessions are written sequentially and never modified in place. JSONL's append model maps directly to this. SQLite's row-based model adds transactional overhead that isn't needed for an append-only workload.

**Recovery is simpler.** JSONL recovery is "scan the file, ignore the last line if it doesn't parse." SQLite recovery can require WAL reconstruction, page repair, or backup restore. For a tool developers run on laptops, simpler recovery matters.

**SQLite as index gives us the best of both.** Fast queries for the API (list sessions, get session metadata) come from SQLite. Durability and human access come from JSONL. The two are not in conflict.

## Tradeoffs accepted

- JSONL cannot be efficiently queried without the SQLite index. Querying a session without `state.db` requires a full file scan.
- Sessions cannot be updated or deleted in-place (only `session.ended` appended, and the file left in place). Deletion requires removing the file and updating SQLite.
- Index and JSONL can diverge if a crash happens between a JSONL write and the index update. Mitigated by: index is rebuilt on startup when inconsistency is detected.

## Alternatives considered

**SQLite only.** Rejected. Binary format, no meaningful git diff, requires tooling to inspect, harder crash recovery.

**Structured log files (one file per event type).** Rejected. Splits a single session across multiple files, making it harder to trace a complete timeline.

**PostgreSQL or another server database.** Rejected. Adds an external dependency that violates the self-contained, zero-dependency-for-development principle.
