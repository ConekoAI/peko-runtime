# ADR-005: Bus-only A2A — no direct agent calls

**Date:** 2026-03-15
**Status:** Accepted
**Deciders:** Core team

## Context

When building multi-agent teams, agents need to communicate. The obvious implementation would let agent A directly call agent B's session endpoint (essentially a remote procedure call). The alternative is requiring all inter-agent communication to pass through a shared message bus.

## Decision

All agent-to-agent communication within a team must go through the team event bus. There is no mechanism in the runtime for agent A to directly invoke agent B. The `sessions_send` tool is explicitly blocked when the target belongs to a team peer. Attempting direct A2A bypasses returns `cross_agent_send_forbidden`.

## Reasoning

**Observability is non-negotiable for debugging multi-agent systems.** Direct calls are invisible — they don't appear in session history, can't be monitored, can't be replayed. Every bus message is written as `a2a.sent` / `a2a.received` in both the sender's and receiver's session. When a multi-agent system misbehaves (and they always do), you need a complete message trace to understand what happened.

**Decoupling enables independent scaling and replacement.** If agent A calls agent B directly, A must know B's instance ID. If B is scaled to three instances, A must implement load balancing. If B is replaced by B-v2, A must be updated. With the bus, A sends to a topic; the bus handles routing, load balancing, and receiver identity. Agents are genuinely decoupled.

**Backpressure is automatic with a bus.** If agent B is busy processing, direct calls would either block A or drop messages. A bus buffers messages and delivers them when B is ready. This is critical for coordinator-worker patterns where many workers may simultaneously try to return results to one coordinator.

**Synchronous A2A is still possible via bus.** An agent that needs a synchronous response sends a `Task` message and waits for the `TaskResult`. This is functionally equivalent to an RPC but remains observable, logged, and decoupled.

## Tradeoffs accepted

- Higher latency for A2A than direct calls. For in-memory bus, the overhead is sub-millisecond and acceptable. For Redis/NATS, round-trip latency is higher but network round-trips were already implied.
- Bus adds a concept that developers unfamiliar with message-passing must learn.
- `sessions_send` is still available for human-to-agent and tooling use cases. The enforcement boundary requires careful documentation to avoid confusion.

## Alternatives considered

**Direct HTTP calls between agents.** Rejected. Invisible to observability, requires agents to know each other's addresses, no backpressure, tight coupling.

**Shared memory / shared state object.** Rejected. Race conditions, no audit trail, no support for multi-process deployments.

**Allow direct calls with optional logging.** Rejected. "Optional" logging will be omitted in production exactly when it is most needed. Consistency is more important than flexibility here.
