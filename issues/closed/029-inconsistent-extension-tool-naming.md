# Inconsistent Extension Tool Naming Convention

## Status

**RESOLVED** — Implemented Option C (check by `extension_id`).

## Problem

Different extension types used inconsistent naming conventions for their registered tools, which caused a whitelist matching bug (fixed in commit `0677c09` as a band-aid).

### Previous State

| Extension Type | `ExtensionId` | Tool Name | `ext enable` adds... | Matches whitelist? |
|---|---|---|---|---|
| Built-in | `builtin:tool:shell` | `shell` | `shell` | ✅ exact match |
| Universal | `universal:calculator` | `calculator` | `calculator` | ✅ exact match |
| MCP | `mcp:standard-echo` | `mcp:standard-echo:echo` | `standard-echo` | ❌ mismatch |

MCP was the **only** extension type that namespaced its tool names (`mcp:{server}:{tool}`). This forced the whitelist checker (`ExtensionConfig::is_extension_enabled`) to special-case `mcp:` prefix parsing to extract the server name and match it against the bare extension ID.

### The Band-Aid Fix (removed)

Commit `0677c09` added MCP-specific parsing to `src/types/agent.rs`:

```rust
if let Some(rest) = name.strip_prefix("mcp:") {
    if let Some(server_name) = rest.split(':').next() {
        if pattern.eq_ignore_ascii_case(server_name) {
            return true;
        }
    }
}
```

This was fragile: any new extension type with a different naming scheme would need another special case.

## Solution Implemented

**Option C: Check by `extension_id` instead of tool name string.**

### How It Works

1. **`ToolRegistry` stores `extension_id` per tool** — a new `tool_owners: HashMap<String, ExtensionId>` field maps each registered tool name to its owning extension's canonical ID.

2. **Whitelist stores canonical extension IDs** — instead of bare tool names, the `enabled` list contains IDs like `builtin:tool:shell`, `mcp:standard-echo`, `universal:calculator`.

3. **`is_tool_enabled` looks up the owner** — when a tool execution is requested, `ToolRegistry::is_tool_enabled(tool_name)` looks up the tool's `extension_id` in `tool_owners` and checks if *that ID* is in the whitelist. No tool-name parsing required.

4. **`ExtensionConfig::is_extension_enabled` is simplified** — removed the MCP band-aid. Now does only exact match + wildcard support. It receives canonical extension IDs, not tool names.

5. **`ext enable` stores canonical IDs** — the IPC handler builds the canonical extension ID (`builtin:tool:{name}` for built-ins, the raw ID for others) before writing to the agent config.

### New State

| Extension Type | `ExtensionId` | Tool Name | Whitelist Entry | Match Logic |
|---|---|---|---|---|
| Built-in | `builtin:tool:shell` | `shell` | `builtin:tool:shell` | exact `extension_id` |
| Universal | `universal:calculator` | `calculator` | `universal:calculator` | exact `extension_id` |
| MCP | `mcp:standard-echo` | `mcp:standard-echo:echo` | `mcp:standard-echo` | exact `extension_id` |

All three types now use the **same** matching logic: look up the tool's owner `extension_id`, check if that ID is in the whitelist.

## Files Changed

- `src/extension/core/tool_registry.rs` — added `tool_owners` HashMap; `register_tool`/`unregister_tool`/`is_tool_enabled` updated
- `src/extension/core/registry.rs` — `register_tool` now passes `extension_id` to `ToolRegistry::register_tool`
- `src/types/agent.rs` — `ExtensionConfig::is_extension_enabled` simplified (MCP band-aid removed); default whitelist uses canonical IDs; `get_extension_settings` uses bare name fallback for built-in per-tool settings
- `src/runtime/tool_runtime.rs` — daemon default whitelist uses canonical extension IDs
- `src/ipc/server.rs` — `ExtensionEnable`/`ExtensionDisable` handlers store canonical extension IDs in agent config
- `src/common/config_path.rs` — test updated for canonical IDs

## Benefits

1. **Single source of truth restored** — `ExtensionConfig` no longer knows about `mcp:` prefixes.
2. **Future-proof** — A new extension type (gateway, custom adapter, etc.) works automatically as long as it registers with a canonical `extension_id`.
3. **Semantically correct** — `ext enable standard-echo` now means "enable the extension whose ID is `mcp:standard-echo`", which is exactly what it says.
4. **No special cases** — One generic matching rule for all extension types.

## Breaking Change

Agent configs that previously whitelisted bare tool names (e.g., `enabled = ["shell"]`) must be updated to use canonical extension IDs (`enabled = ["builtin:tool:shell"]`). The default config generated by `AgentConfig::default()` has been updated accordingly.
