# ADR-002: Image / instance as a first-class distinction

**Date:** 2026-03-15
**Status:** Accepted
**Deciders:** Core team

## Context

Early designs treated "agent" as a single concept — the directory you write and the thing that runs were the same object. This caused confusion: if you edit a file in a running agent's directory, does the running agent see the change? Does sharing an agent mean sharing its conversation history?

## Decision

The system distinguishes two first-class concepts:

- **Agent image**: a frozen, versioned, immutable snapshot of an agent's definition. Equivalent to a Docker image. Never contains session history.
- **Agent instance**: a living runtime derived from an image. Has its own session history, workspace, and cron registrations. Equivalent to a Docker container.

These are separate things in every layer: data model, API, CLI verbs, filesystem layout, and mental model.

## Reasoning

**The analogy to Docker is immediately legible.** Developers who know Docker already understand image/container. Reusing that mental model reduces onboarding friction. The concepts map cleanly: `pekobot build` ≈ `docker build`, `pekobot run` ≈ `docker run`, `pekohub.com` ≈ Docker Hub.

**Sharing must not imply shared state.** If two users pull the same agent image and run it, they must get independent conversation histories. Without the image/instance split, this requires awkward per-user copy semantics at the file level. With the split, it is natural: same image, different instances.

**Immutability enables reproducibility.** An agent image at a fixed digest is bit-for-bit reproducible. This matters for research workflows (reproduce the exact agent that generated a result) and for audit (prove which agent definition was running at a given time).

**Editing a running agent is a development-only operation.** Watch mode (`--watch`) is the explicit affordance for "I am actively developing this agent and want live reload." Outside watch mode, modifying source files does not affect running instances. This prevents accidental live modification in production.

## Tradeoffs accepted

- More concepts to learn upfront. A user who just wants to "run their script" must understand that `pekobot run` creates an instance from an image, not "runs the directory."
- `pekobot commit` (promoting instance changes back to an image) is a necessary but complex operation that has no direct filesystem analogy.

## Alternatives considered

**Single "agent" concept (directory = running thing).** Rejected. Conflates definition and state, makes sharing ambiguous, makes reproducibility impossible without external conventions.

**Implicit copy-on-run (like some VM systems).** Rejected. Hides what's actually happening, makes debugging harder, and is surprising when the "copy" diverges from the original.
