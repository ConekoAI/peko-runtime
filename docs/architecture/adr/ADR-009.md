# ADR-009: `sessions_send` blocked for agent-to-agent within a team

**Date:** 2026-03-16
**Status:** Accepted
**Deciders:** Core team

## Context

`sessions_send` lets a caller send a message to any session by ID. It was designed for human-to-agent and tooling use cases (e.g. a CI script injecting a message into a running session). The question was whether agents should be allowed to use it to message peer agents in the same team.

This was debated because some use cases feel natural as direct session messages (e.g. a coordinator sending a formatted report to a writer agent's session). The bus alternative requires using `Task`/`TaskResult` message types, which is slightly more verbose.

## Decision

`sessions_send` is blocked when the target session belongs to an agent instance in the same team as the caller. The runtime enforces this at the tool invocation layer, returning `cross_agent_send_forbidden`. The error message explains that `A2A` via the event bus should be used instead.

## Reasoning

**It would silently bypass the bus.** ADR-005 established that all A2A communication must go through the bus for observability, decoupling, and backpressure. `sessions_send` is a direct session injection that skips the bus entirely. If agents could use it freely, the bus-only invariant would be an illusion — a convention, not an enforcement.

**No `a2a.sent` event would be written.** Bus messages appear as `a2a.sent` / `a2a.received` in session history, making them auditable. A `sessions_send` from one agent to another would appear only as a `user.message` with no indication it came from another agent. The audit trail would be misleading.

**Backpressure would be lost.** The bus buffers messages when a consumer is busy. `sessions_send` injects directly into a session queue. A coordinator flooding a writer with `sessions_send` calls while the writer is mid-turn could overwhelm it. The bus handles this gracefully; direct injection does not.

**The bus is not actually more verbose.** Sending a `Task` message to a specific agent via `Direct` addressing is one `a2a.sent` call. The verbosity concern was overstated.

**The legitimate use case for `sessions_send` remains intact.** Human operators, CI scripts, the TUI, and external integrations all have valid reasons to inject messages into sessions. None of these are team-peer agents. The restriction precisely targets the problematic case without affecting legitimate uses.

## Tradeoffs accepted

- Agents that want to "notify" a specific peer agent must use the bus even if the message is simple. The overhead is small but not zero.
- The enforcement requires the runtime to know which instances are in which team at tool invocation time — a runtime lookup that adds a small amount of coupling between the tool layer and the team model.

## Alternatives considered

**Allow it but log a warning.** Rejected. Warning-only enforcement is not enforcement. It will be ignored in production and discovered too late.

**Allow it with an explicit `bypass_bus: true` flag.** Rejected. Creates a footgun with a label on it. There is no legitimate reason for an agent to bypass the bus for peer communication.

**Block at the team level via policy (not at runtime).** Rejected for Phase 1. Could be reconsidered in Phase 2 under control plane policies, but hardcoding it in Phase 1 ensures the invariant holds before the control plane exists.
