# ADR-003: Instances pin to digest, not tag

**Date:** 2026-03-15
**Status:** Accepted
**Deciders:** Core team

## Context

When creating an instance from `pekohub.com/agents/researcher:v2.5`, the runtime could pin the instance to either the human-readable tag (`researcher:v2.5`) or the content-addressed digest (`sha256:a3b5c7...`). If pinned to the tag, pulling an updated `researcher:v2.5` could silently change the behaviour of existing instances. If pinned to the digest, behaviour is frozen at creation time.

## Decision

Instances always pin to the exact image digest at creation time. The tag used to create the instance is recorded for human reference, but the instance's identity is the digest. Pulling a newer image with the same tag has no effect on running or stopped instances. Upgrade is explicit: `pekobot instance upgrade <id> --to <image:tag>`.

## Reasoning

**Silent behaviour change is the worst failure mode for agents.** An agent that was working correctly and then silently changes behaviour — because someone pushed a new image with the same tag — is extremely difficult to debug. The operator has no signal that something changed. With digest pinning, the agent's behaviour is frozen until an explicit upgrade decision is made.

**Reproducibility requires digest-level identity.** "The agent that generated this output" must be precisely identifiable. A tag is mutable; a digest is not. Audit trails, research reproducibility, and production rollback all require digest-level identity.

**Explicit upgrade is a forcing function for intent.** Requiring `pekobot instance upgrade` means someone must consciously decide "I want this instance to run the new image." This surfaces upgrade as a deliberate action with observable consequences, not a side effect of a `pull`.

**This is how Docker works in production.** Production Docker deployments routinely pin to digests for exactly this reason. We are applying the same discipline by default rather than leaving it to operators to remember.

## Tradeoffs accepted

- More friction to update instances. A developer who pushes a new image must explicitly upgrade running instances, even in development. Partially mitigated by watch mode for local development.
- The tag displayed in `pekobot ps` can become stale (pointing to an older digest than the current `latest`). The digest is always accurate; the tag is informational.
- `pekobot instance upgrade` must handle in-progress sessions gracefully (wait for current turn to complete, then apply).

## Alternatives considered

**Pin to tag (always use latest digest for the tag).** Rejected. Silent behaviour change, no reproducibility, violates the principle of least surprise.

**Let the user choose (flag at `pekobot run`)**.** Rejected. Digest pinning is always correct for production. Making it optional means it will be wrong in production whenever a developer forgets the flag.
