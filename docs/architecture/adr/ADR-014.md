# ADR-014: Simplified Shell Tool and Filesystem Permissions (All-or-Nothing Model)

**Date:** 2026-03-25
**Status:** Accepted
**Deciders:** Core team

## Context

The current `process` tool implements fine-grained security restrictions:
- Blocks shell execution (sh, bash, zsh, cmd, powershell)
- Requires command + args array (no shell string parsing)
- Strips sensitive environment variables
- Validates working directory is within sandbox
- Prevents path traversal attacks

Similarly, the `filesystem` tool enforces:
- Workspace-only access by default
- Path traversal blocking
- Security policy checks on every operation

This security model adds significant complexity (~600+ lines of validation code) while still being circumventable via custom tools or MCP servers. Users who need full shell capabilities must work around the restrictions, while users who want safety must carefully configure the sandbox.

## Decision

**Adopt an all-or-nothing permission model for core tools.**

1. **Rename `process` to `shell`** - The tool now accepts a raw shell string and executes it via the system shell (bash/sh on Unix, cmd/powershell on Windows).

2. **Remove all sandboxing restrictions** - When enabled, the tool has full access:
   - No shell blocking
   - No command/args separation (accepts raw string)
   - No environment variable filtering
   - No working directory restrictions
   - Full shell features: pipes, redirection, shell builtins, etc.

3. **Same treatment for `filesystem`** - Remove workspace-only restrictions, allow full filesystem access when enabled.

4. **Security boundary is tool enablement** - Either:
   - Tool is **disabled** → No access at all
   - Tool is **enabled** → Full access (no restrictions)

5. **Fine-grained access control delegated to MCP** - Users needing restricted access (e.g., "only allow git commands" or "only allow /home/user workspace") can use MCP tool addons or custom tools.

## New Tool API

### `shell` tool (replaces `process`)

```json
{
  "command": "ls -la | grep foo > output.txt",
  "timeout_ms": 30000,
  "async": false,
  "cwd": "./subdir"
}
```

**Parameters:**
- `command` (string, required): Raw shell command to execute
- `timeout_ms` (integer, optional): Timeout in milliseconds (default: 120000)
- `async` (boolean, optional): Execute asynchronously (default: false)
- `cwd` (string, optional): Working directory (default: agent workspace)

**Response:**
```json
{
  "exit_code": 0,
  "stdout": "...",
  "stderr": "...",
  "success": true
}
```

### `filesystem` tool

Remains functionally the same but without sandbox restrictions. Can read/write any path the OS user has permission for.

## Configuration

```toml
[tools]
# Enable/disable core tools (default: true)
enable_shell = true      # Full shell access
enable_filesystem = true # Full filesystem access
enable_apply_patch = true
enable_cron = true
```

**Note:** Both default to `true` for backward compatibility. The security model will be revisited in a future ADR.

## Reasoning

**Simplicity over false security.**

The previous sandbox model created a false sense of security. A determined agent could:
- Write a script to disk and execute it
- Use `find` with `-exec` for arbitrary command execution
- Exploit tool ordering to escape restrictions
- Use MCP servers for unrestricted access

Meanwhile, legitimate use cases suffered:
- Pipes and redirection required workarounds
- Shell builtins (echo, cd, source) were unavailable
- Environment manipulation was restricted
- Cross-platform scripting was difficult

**Clear mental model.**

Users understand "this tool is powerful, I can disable it if I don't need it." The previous model required understanding subtle sandbox rules (what's a "shell"? why can't I use `bash -c`? why is my env var stripped?).

**Aligned with extensibility philosophy.**

Peko's design delegates specialized behavior to MCP servers. Restricted shell access is a specialized need ("only git commands", "only in this directory"). Users with those needs should use MCP addons or custom tools rather than expecting the core tool to be configurable.

**Reduced maintenance burden.**

Removing sandbox logic eliminates:
- Shell detection and blocking code
- Environment variable filtering rules
- Path traversal validation
- Working directory sandboxing
- Cross-platform shell compatibility issues

## Consequences

### For Users

- **More powerful by default** - Full shell access when `enable_shell = true`
- **Simpler API** - Just pass a command string, no args array
- **Less protection from accidents** - `rm -rf /` will execute if typed
- **Clearer security boundary** - Know that enabled = full access

### For Implementation

**Removed from `shell` tool:**
- `BLOCKED_SHELLS` constant and `is_blocked_shell()` function
- `SENSITIVE_ENV_PATTERNS` and `filter_env_vars()` function
- `validate_cwd()` sandboxing function
- `ProcessArgs` struct (replaced with simpler struct using `command` string)
- Shell injection protection documentation

**Simplified behavior:**
- Pass `command` directly to `sh -c` (Unix) or `cmd /c` (Windows)
- No environment filtering (inherit full env)
- `cwd` is passed directly to shell, no validation
- Remove `args` parameter entirely

**Same treatment for `filesystem`:**
- Remove `SecurityPolicy` from tool
- Remove workspace-only checks
- Allow any OS-permitted path

### Security Posture

This is a **deliberate regression in default safety** in exchange for simplicity and capability. We acknowledge:

- Accidental destructive commands are more likely to succeed
- Malicious prompts have more attack surface
- Users must rely on external tools for sandboxing

This is acceptable for the current target audience (technical users, local-first deployment) but will need revisiting for:
- Multi-tenant deployments
- Non-technical users
- Production automation scenarios

## Alternatives Considered

**Keep current sandbox model.** Rejected. Complexity without sufficient security benefit. Users work around it anyway.

**Add configuration for restriction levels.** Rejected. Violates the "all-or-nothing" simplicity goal. Fine-grained permissions belong in MCP addons.

**Default to disabled.** Considered but rejected for now. Would be a breaking change for existing users. Will revisit in security model ADR.

## Implementation Notes

### Shell Selection

**Unix:** Use `/bin/sh -c` (POSIX-compliant, always available)
**Windows:** Use `cmd /c` (default shell, no PowerShell dependency)

Platform detection:
```rust
#[cfg(unix)]
const SHELL: &str = "/bin/sh";
#[cfg(windows)]
const SHELL: &str = "cmd";
#[cfg(unix)]
const SHELL_ARG: &str = "-c";
#[cfg(windows)]
const SHELL_ARG: &str = "/c";
```

### Migration

- Rename `ProcessTool` → `ShellTool`
- Rename `enable_process` → `enable_shell` (accept both for backward compat)
