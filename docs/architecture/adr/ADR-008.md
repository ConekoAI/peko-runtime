# ADR-008: Heavy tools as MCP servers, not built-ins

**Date:** 2026-03-16
**Status:** Accepted
**Deciders:** Core team

## Context

Early versions of Peko included `web_search`, `fetch`, `http`, `browser`, and `memory` as built-in tools. These were migrated to standalone MCP servers. ADR-007 explains why built-ins are compiled-in; this ADR explains why these specific tools were moved out.

> **Update:** The built-in MCP servers (`mcp-web`, `mcp-browser`, `mcp-memory`) have been removed from the main repository. They should be maintained as separate external projects or obtained from the community MCP servers ecosystem.

## Decision

Tools that require heavy external dependencies (browser automation, HTTP clients, vector databases) are implemented as MCP servers, not built-in tools. The runtime ships no HTTP client, no browser, and no vector store. Agents that need these declare them as MCP capability dependencies.

## Reasoning

**Binary size and startup time.** A Playwright browser, an HTTP client with TLS, and a vector embedding library are collectively tens of megabytes. Compiling them into the runtime binary defeats the single-binary deployment goal and increases cold start time. As MCP servers, they are separate processes started only when needed.

**Dependency isolation.** Browser automation depends on Playwright/Chromium. Vector memory depends on an embedding model or vector database client. These have their own dependency trees, update cycles, and security surface. Keeping them in separate processes means a vulnerability in the browser library doesn't affect the core runtime.

**Optional by design.** Most agents don't need a browser. Making browser automation an opt-in MCP server means agents that don't use it pay nothing — no startup cost, no memory usage, no attack surface. Built-in inclusion would impose this cost on everyone.

**MCP is the right abstraction.** The Model Context Protocol was designed precisely for this use case: giving LLMs access to external tools via a standardised process interface. Using it for heavy tools validates the protocol and makes these tools interoperable with any MCP-compatible system, not just Peko.

**Shared MCP servers multiply the value.** A single browser automation process shared across all agents in a team (via the shared services fabric) is dramatically more efficient than one browser process per agent. This sharing model is natural for MCP servers and awkward for built-in tools.

## Tradeoffs accepted

- Agents that need web access must declare an MCP dependency. This is slightly more setup than having `web_search` available automatically.
- MCP server startup time adds to team deploy time. Mitigated by: MCP servers are started once per team and shared.
- A broken MCP server affects all agents that depend on it. Mitigated by: MCP server crash recovery (ADR not written yet, but REQ-RL-004 covers the requirement).

## Alternatives considered

**Built-in HTTP client (no browser).** `fetch` and `http` as built-ins with no external dependencies. Still under consideration for a future minimal networking built-in, but deferred to avoid scope creep in Phase 1. `mcp-web` covers all networking needs for now.

**Bundling MCP servers in the main binary.** Rejected. Defeats dependency isolation. The whole point is that MCP servers can be updated independently of the runtime.
