# ADR-004: Single daemon process as sole control point

**Date:** 2026-03-15
**Status:** Accepted
**Deciders:** Core team

## Context

CLI tools typically either operate directly on files (e.g. `git`) or talk to a background daemon (e.g. `docker`). For Pekobot, we needed to decide whether the CLI manipulates agent state directly or through a daemon.

Direct manipulation would mean the CLI reads and writes `.peko/` directly. A daemon means the CLI issues HTTP requests and the daemon does the actual work.

## Decision

All runtime operations are mediated by a single local daemon process (`peko daemon`). The CLI, TUI, Web UI, and external integrations all talk to the daemon via IPC (UDP on Windows, Unix domain sockets on Unix). No client has direct access to agent internals.

## Reasoning

**Running agents are stateful processes.** Unlike `git` repositories, running agent instances have in-memory state: the active session, tool call receipts, in-flight async tasks, cron schedules. There is no way for a CLI command to safely interoperate with this state without going through the process that owns it.

**The daemon is the natural place for shared state.** The event bus, shared MCP server processes, reference counting for shared services — all of these require a single process with a consistent view of what is running. A CLI that manipulates files directly cannot coordinate with other CLI invocations or with running instances.

**Ollama proved the model works.** The Ollama architecture (daemon on port 11434, CLI and Web UI as clients) is well-understood, easy to explain, and easy to integrate with. We use a similar but distinct port (11435) to avoid conflicts while keeping the approach familiar to developers who have used Ollama.

**Third-party integrations become trivial.** Any language that can make HTTP requests can integrate with Pekobot. The daemon's HTTP API is a stable, versioned interface. Without a daemon, third-party integrations would need to understand and manipulate the `.peko/` filesystem layout, which is not a stable API.

**Single responsibility for security.** The daemon is the sole enforcer of capability grants, sandbox rules, and credential isolation. With direct file manipulation, each client would need to enforce these independently — and inevitably inconsistently.

## Tradeoffs accepted

- Users must start the daemon before running any CLI commands. `peko daemon start` is a required step. Mitigated by: `peko daemon start` is a required step before running other commands.
- The daemon is a single point of failure. If it crashes, nothing works. Mitigated by: fast restart, state rebuilt from JSONL on restart.
- Development without a daemon (e.g. in CI without a background process) is harder. Mitigated by: `peko daemon start --foreground` keeps the daemon in the foreground.

## Alternatives considered

**Direct file manipulation (git-style).** Rejected. Cannot manage in-memory state, cannot coordinate concurrent operations, cannot support a stable integration API.

**Per-operation subprocess (no persistent daemon).** Rejected. Cannot maintain event bus, cannot hold MCP server processes open, cannot support streaming responses to multiple clients simultaneously.
