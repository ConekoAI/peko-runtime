# ADR-015: Unified Event-Driven Architecture for Streaming

**Date:** 2026-03-25
**Status:** Accepted
**Deciders:** Core team

## Context

The current architecture has a fundamental mismatch between how streaming and blocking execution are handled:

1. **SRP Violation:** `MessageService` has two methods (`send_message` and `send_message_streaming`) that implement the same business logic but differ only in presentation - which is not the service layer's concern.

2. **DRY Violation:** Both blocking and streaming paths duplicate:
   - Session resolution logic
   - Agent cold-start logic  
   - Session persistence logic
   - Event handling (just processed differently)

3. **Inconsistency:** The HTTP API supports streaming (SSE) but the CLI does not, despite both using the same underlying `AgenticLoopV4` which always emits events.

4. **Maintenance Burden:** Adding a new presentation mode (e.g., WebSocket, file output) requires modifying the service layer, violating the Open/Closed principle.

## Decision

**Adopt a unified event-driven architecture where streaming is the fundamental execution mode, and blocking is "streaming with collection."**

### Core Principle

The service layer emits events; the channel layer decides how to consume them:

```
┌─────────────────────────────────────────────────────────────┐
│                    PRESENTATION LAYER                       │
│  (Commands/Web Routes - User-facing entry points)          │
└───────────────────────┬─────────────────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────────────────┐
│                    CHANNEL LAYER                            │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐ │
│  │  CliChannel │  │ HttpChannel │  │  WebSocketChannel   │ │
│  │             │  │  (SSE)      │  │                     │ │
│  │ • Blocking  │  │ • Streaming │  │ • Real-time         │ │
│  │ • Streaming │  │             │  │ • Bidirectional     │ │
│  └──────┬──────┘  └──────┬──────┘  └──────────┬──────────┘ │ │
└─────────┼────────────────┼────────────────────┼────────────┘
          │                │                    │
          └────────────────┴────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                    SERVICE LAYER                            │
│              MessageService::send_message()                 │
│         (Single unified method - always emits events)       │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                    ENGINE LAYER                             │
│              AgenticLoopV4::run_with_events()               │
│         (Always streaming, returns EventStream)             │
└─────────────────────────────────────────────────────────────┘
```

### Architecture Changes

#### 1. Service Layer (SRP Compliance)

**Before:** Two methods with duplicated logic
```rust
impl MessageService {
    pub async fn send_message(&self, request: MessageRequest) -> Result<MessageResult>;
    pub async fn send_message_streaming(&self, request: MessageRequest) -> Result<mpsc::Receiver<ChatEvent>>;
}
```

**After:** Single method that returns event stream
```rust
impl MessageService {
    /// Returns a stream of AgenticEvents
    pub async fn send_message(
        &self, 
        request: MessageRequest
    ) -> Result<EventStream>;
}
```

The service layer has no knowledge of:
- Whether output is blocking or streaming
- What transport is used (CLI, HTTP, WebSocket)
- How events are formatted

#### 2. Channel Layer (Presentation Concern)

Extend the `Channel` trait with event processing:

```rust
#[async_trait]
pub trait Channel {
    fn name(&self) -> &str;
    
    /// Process event stream according to channel's needs
    async fn process_stream(
        &self,
        event_rx: mpsc::Receiver<AgenticEvent>,
    ) -> Result<ChannelOutput>;
}
```

**Channel Implementations:**

- `CliChannel::process_stream()` - Two modes:
  - **Interactive:** Print tokens as they arrive (streaming)
  - **Script:** Collect and return final result (blocking)
  
- `HttpChannel::process_stream()` - Convert to SSE events
- `WebSocketChannel::process_stream()` - Convert to WS messages

#### 3. Command Layer (Thin Wrapper)

```rust
pub async fn handle_send(args: SendArgs, ...) -> Result<()> {
    // Create channel with appropriate mode
    let channel = CliChannel::new(&args.agent)
        .with_mode(if args.stream { 
            CliMode::Streaming 
        } else { 
            CliMode::Blocking 
        });
    
    // Use unified service
    let event_stream = message_service.send_message(request).await?;
    
    // Let channel handle presentation
    let output = channel.process_stream(event_stream).await?;
    
    Ok(())
}
```

## Reasoning

**Streaming is the natural abstraction.**

LLM providers always stream tokens internally. Converting to "blocking" is just collecting the stream. Embracing this simplifies the mental model: everything is streaming, consumption varies.

**Proper separation of concerns.**

- **Service layer:** Business logic (load config, resolve session, run agent)
- **Channel layer:** Presentation (how to display/process events)

This aligns with ports-and-adapters architecture - the service is a port, channels are adapters.

**DRY compliance.**

Session resolution, agent cold-start, and persistence happen once in the service layer. No duplication between blocking and streaming paths.

**Open/Closed principle.**

New presentation modes require only a new `Channel` implementation:
- File logging channel? Just implement `Channel::process_stream()`
- Metrics collection channel? Same interface
- Multi-channel broadcast? Compose channels

No service layer changes needed.

**Testability.**

- Test service layer: Inject mock event consumer, verify events emitted
- Test channels: Inject mock event stream, verify output format
- Test end-to-end: Full stack with test doubles

## Consequences

### For Users

- **CLI gains streaming support** - `--stream` flag shows tokens as they arrive
- **Consistent experience** - API and CLI use same execution path
- **Better perceived performance** - See response immediately instead of waiting

### For Implementation

**Files to modify:**
- `src/common/services/message_service.rs` - Unify to single `send_message()` returning `EventStream`
- `src/channels/mod.rs` - Add `process_stream()` to `Channel` trait
- `src/channels/cli.rs` - Implement `process_stream()` with blocking/streaming modes
- `src/commands/send.rs` - Use channel pattern instead of direct service call
- `src/api/routes/chat.rs` - Adapt to use `HttpChannel::process_stream()`

**Files potentially removed:**
- `send_message_streaming()` method (superseded by unified approach)
- Duplicate session resolution logic in streaming path

### Migration Strategy

**Phase 1:** Implement `Channel::process_stream()` for `CliChannel` (blocking mode)
- Lowest risk, maintains current CLI behavior
- Establishes new pattern

**Phase 2:** Refactor `MessageService` to return `EventStream`
- Internal refactoring, no external behavior change
- HTTP API continues working via adapter

**Phase 3:** Add `--stream` flag to CLI
- New feature, backward compatible
- Uses existing streaming infrastructure

**Phase 4:** Refactor HTTP API to use `HttpChannel`
- Consistency across all entry points
- Optional if current SSE implementation works

## Alternatives Considered

**Keep separate blocking/streaming methods.** Rejected. Violates DRY, harder to maintain, encourages divergence between paths.

**Make service layer return final result only.** Rejected. Forces blocking at service layer, streaming must be hacked on top.

**Use callbacks instead of event streams.** Rejected. Callbacks couple producer to consumer; streams allow decoupled composition (multiple channels, transformation, etc.).


### CliChannel Modes

```rust
pub enum CliMode {
    /// Print tokens as they arrive (for interactive use)
    Streaming,
    /// Collect and return final result (for scripts)
    Blocking,
}
```

### Backward Compatibility

HTTP API SSE endpoint remains unchanged externally. Internally it uses `HttpChannel::process_stream()` instead of `send_message_streaming()`.

CLI default behavior remains blocking. `--stream` flag enables streaming mode.
