# ADR-006: In-memory Tokio channels as default bus backend

**Date:** 2026-03-15
**Status:** Accepted
**Deciders:** Core team

## Context

Given ADR-005 (bus-only A2A), the bus needs a default implementation. Options range from in-process channels to embedded message brokers (like NATS embedded mode) to external brokers (Redis Streams, NATS).

## Decision

The default bus backend is in-memory Tokio async channels (`tokio::sync::mpsc` and `broadcast`). Redis Streams and NATS are supported as alternative backends for multi-process and multi-host deployments. The backend is declared in `[shared.bus] backend` in `team.toml`.

## Reasoning

**Zero dependencies for the common case.** The vast majority of Phase 1 Pekobot deployments will be single-node, single-process. Requiring Redis or NATS for a basic team deployment would create an immediate setup barrier. In-memory channels need nothing beyond the Rust standard library and Tokio (which the runtime already depends on).

**Sub-millisecond latency.** In-process channel sends are ~200ns. For the tight coordination loops common in multi-agent systems (coordinator sends task, worker returns result, coordinator sends next task), this latency is essentially free. Redis adds ~1ms per round-trip; NATS adds ~0.5ms. For high-frequency A2A, this compounds.

**Tokio channels are battle-tested.** They have well-understood semantics, excellent documentation, and are already in the dependency tree. An embedded message broker would add complexity and a new failure mode.

**The interface is the same regardless of backend.** The `Bus` trait abstracts over the backend. Switching from in-memory to Redis requires only changing `team.toml`. No agent code changes. The in-memory backend does not expose implementation details that other backends cannot match.

**In-memory is honest about its limitations.** It doesn't persist messages across restarts and only works within a single process. These limitations push operators toward Redis/NATS when they actually need those properties, rather than discovering the limitations in production.

## Tradeoffs accepted

- Messages are lost on daemon crash (in-memory backend). Agents relying on bus messages for critical coordination must either handle re-delivery at the application layer or use Redis/NATS.
- Cannot span multiple processes or hosts with the in-memory backend. This is a deliberate constraint, not an oversight.
- The `broadcast` channel capacity is bounded (default 1024). A slow consumer that falls too far behind will miss messages. Documented as a known limitation.

## Alternatives considered

**NATS embedded.** Appealing (persistence, clustering, zero external dependency via embedding) but adds ~15MB to the binary and a non-trivial startup cost. Deferred to Phase 3.

**Redis Streams as default.** Rejected. Requires Redis to be running before any team can be deployed. Too high a barrier for local development.

**Custom ring buffer implementation.** Rejected. Reinventing a well-solved problem. Tokio channels are correct and fast.
