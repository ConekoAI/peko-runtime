# ADR-010: Cron registrations owned by instance, persisted to disk

**Date:** 2026-03-16
**Status:** Accepted
**Deciders:** Core team

## Context

The `cron` built-in tool lets an agent schedule future tasks from within a session. This raises lifecycle questions: where is the cron state stored? Who owns it? What happens on instance restart, image upgrade, or daemon crash?

Two broad options: store cron registrations in SQLite (alongside the instance state index) or in a separate JSON file in the instance workspace.

## Decision

Cron registrations are stored in `cron.json` in the instance workspace (same directory as `sessions/`). They are owned by the instance, not the image. Writes are atomic. On restart, the daemon reloads `cron.json` and reschedules all active jobs. On image upgrade, registrations are preserved and hook-defined jobs from the new image are merged in.

## Reasoning

**Cron state is instance-owned, not image-owned.** An agent image defines hook-level cron schedules in `config.toml` (which are static). But `cron` tool calls from within sessions create dynamic registrations that are part of the instance's living state — not its definition. Separating them is conceptually correct and avoids a case where upgrading an image accidentally removes dynamically-created schedules.

**`cron.json` is human-readable and auditable.** Like sessions, cron state benefits from being inspectable without tooling. An operator can `cat cron.json` to see all scheduled jobs, their next run times, and their run counts. A SQLite row is not directly inspectable.

**Atomic writes prevent corruption.** Cron state is small (typically <1KB) and written infrequently (only on register/cancel). Atomic write (`.tmp` + rename) is sufficient and reliable. There is no need for the transactional overhead of SQLite for this use case.

**Decoupling from SQLite simplifies recovery.** If `state.db` is deleted, cron state must still be recoverable. Storing it in `cron.json` means cron recovery is independent of the SQLite rebuild process.

**Clear ownership prevents cross-instance interference.** Storing cron in a per-instance file makes the ownership boundary unambiguous. Two instances of the same image have separate `cron.json` files with separate schedules. There is no shared cron state.

## Tradeoffs accepted

- `cron.json` is a separate file that operators must be aware of (alongside `sessions/` and `workspace/`). Added to `.gitignore` template.
- On image upgrade, `orphaned` jobs (from hooks no longer in the new `config.toml`) are not automatically removed. Operators must explicitly cancel them. This is intentional — automatic removal could delete jobs the agent created dynamically with the same label as a config-defined hook.
- The daemon must scan all instance workspaces on startup to reload cron state. For 50+ instances, this is still fast (small JSON files, parallel reads), but it is a startup cost.

## Alternatives considered

**Store cron in SQLite `state.db`.** Rejected. Couples cron recovery to SQLite recovery, not human-readable, harder to diff.

**Store cron in the session JSONL.** Rejected. Sessions are conversation history; cron is runtime configuration. Mixing them would require scanning the entire session to reconstruct cron state on restart.

**In-memory only (lost on restart).** Rejected. Cron jobs are often the agent's primary activation mechanism (scheduled reports, daily digests). Losing them on crash would silently break the agent's purpose.
