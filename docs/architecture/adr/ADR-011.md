# ADR-011: Filesystem-first — packaging is optional

**Date:** 2026-03-15
**Status:** Accepted
**Deciders:** Core team

## Context

Container systems like Docker require building an image before running it. This works well for distribution but adds friction during development (edit → build → run vs. just edit → run). We needed to decide whether Pekobot requires packaging before execution.

## Decision

Packaging is optional and only required for distribution. Agents run directly from a filesystem directory without any build step. `pekobot run ./my-agent/` works immediately after creating `config.toml`. Packaging (`pekobot build`) is a separate step taken only when distributing.

## Reasoning

**Development friction is the enemy of adoption.** If every code change requires a rebuild before testing, developers will either batch changes (slower feedback loop) or find the tool frustrating and abandon it. The ability to `pekobot run ./agent/ --watch` and see changes in under 2 seconds is a direct consequence of filesystem-first execution.

**Most agents are never distributed.** Personal assistants, internal automation, development tools — the majority of agents are built and used by the same person or team. For these, the registry and packaging system add ceremony with no benefit. Packaging should be opt-in.

**Packaging and running from source must be equivalent.** The filesystem loader and the package loader share the same loading logic. Running from a package is not "production mode" with different behaviour — it is identical to running from source, just with the files served from the local package cache rather than the filesystem. This equivalence is testable and enforced.

**The development → distribution workflow is clean.** `pekobot run ./agent/` during development. `pekobot build && pekobot push` when ready to share. The step is explicit and optional, not implicit and required.

## Tradeoffs accepted

- Running from source means the runtime must handle the case where source files change under it (watch mode). This is additional complexity in the filesystem watcher.
- Without packaging, there is no content-addressable identity for a "source agent." You cannot reference a source agent by digest. This is acceptable because source agents are by definition mutable.
- If a developer moves or deletes the source directory, the "image" disappears. There is no local registry entry for source agents. Build an image to make it persistent in the local cache.

## Alternatives considered

**Require build before run (Docker-style).** Rejected. Unacceptable development friction. The build step adds 5–30 seconds with no benefit for local development.

**Auto-build on run (implicit build).** Rejected. Hides what's happening, produces surprise when the build fails or takes time. Explicit is better than implicit.

**Always run from filesystem, no packaging.** Rejected. Distribution requires a stable, portable artifact format. Sharing a filesystem directory is fragile (depends on paths, relative references, missing tools). Packaging solves a real distribution problem.
